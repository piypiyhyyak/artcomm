# Сайт АНО «АртКомм»

Рабочий репозиторий сайта на `React + Vite` с встроенной CMS-панелью.

## 1. Что это за проект

Проект состоит из двух частей:

- публичный сайт (`/`, `/about`, `/documents`),
- админ-панель (`/admin` или `?page=admin`) для редактирования контента.

CMS хранит данные в двух состояниях:

- `draft` (черновик),
- `published` (опубликованная версия).

Публикация переносит текущий `draft` в `published`.

## 2. Архитектура в двух словах

- `Vite` поднимает фронтенд и в dev-режиме прокидывает API через middleware-плагин.
- API CMS реализован в `cmsApiCore.js` (Node, без отдельного тяжелого backend-framework).
- В проде API может работать как отдельный процесс через `cmsServer.js`.

Поток данных:

1. Админ редактирует поля/медиа.
2. Изменения сохраняются в `draft`.
3. API автоматически создает snapshot-версию (резервную копию).
4. По кнопке «Опубликовать» `draft` становится `published`.

## 3. Структура репозитория

```text
.
├─ src/
│  ├─ AdminApp.jsx                 # админ-панель
│  ├─ HomeApp.jsx                  # главная страница
│  ├─ AboutApp.jsx                 # /about и /documents
│  ├─ main.jsx                     # определение типа страницы и mount
│  ├─ styles.css                   # общие стили сайта и админки
│  ├─ initSite.js                  # рендер и интерактив главной
│  ├─ homeMarkup.js                # шаблоны/разметка главной
│  ├─ aboutMarkup.js               # шаблоны/разметка страницы сведений
│  └─ cms/
│     ├─ defaultContent.js         # дефолтная структура контента/ролей
│     └─ storage.js                # клиентская работа с CMS API
├─ public/
│  ├─ assets/                      # статические файлы (изображения, pdf и т.п.)
│  ├─ about/index.html             # прямой вход /about
│  ├─ documents/index.html         # прямой вход /documents
│  └─ privacy/index.html           # политика обработки ПДн
├─ cmsApiCore.js                   # серверная логика CMS API
├─ cmsServer.js                    # standalone HTTP-сервер для CMS API
├─ vite.adminApiPlugin.js          # middleware API для локальной разработки
├─ vite.config.js
└─ deploy/systemd/
   └─ artcomm-cms.env.example      # пример env-файла с секретами
```

## 4. Роутинг

- `/` — главная
- `/about` — сведения об организации
- `/documents` — документы (about-раздел)
- `/admin` — админ-панель
- `/?page=admin` — альтернативный вход в админку

## 5. Медиа и документы

### Важно по видео

- `*.mp4` в Git **не храним** (есть правило в `.gitignore`).
- Для продакшена используем внешний файл по пути:
  - `/assets/gimn-ed-zy9mar.mp4`
- Не переключать на несуществующий `...-lite.mp4`.

### Добавление файлов

Через админку файл загружается в `public/assets/...` через API `/api/cms/upload`.

При замене документа/картинки ссылка в конкретной карточке меняется на **новый путь**. Старый файл не переписывается «по месту», это защищает другие карточки, если они ссылаются на прежний файл.

## 6. Версионный контроль в админке

В админке есть отдельный раздел **«Версионный контроль»**.

Что делает система:

- автоматически создает версию после сохранения черновика,
- автоматически создает версию после публикации,
- создает версию после отката (чтобы история была непрерывной).

Ограничения по ролям:

- смотреть список версий могут авторизованные пользователи,
- откатывать и удалять версии может **только администратор**.

Лимит хранения:

- `ARTCOMM_CMS_VERSION_LIMIT` (по умолчанию `30`).
- старые версии удаляются автоматически при достижении лимита.

## 7. Роли в CMS

Поддерживаемые роли:

- `Администратор`
- `Редактор`
- `Наблюдатель`

Права:

- Администратор: полный доступ, включая пользователей и откаты версий.
- Редактор: редактирование контента и публикация.
- Наблюдатель: просмотр без прав публикации/редактирования.

## 8. Локальный запуск

### Быстрый старт

```bash
npm install
npm run dev
```

По умолчанию Vite поднимается на `http://127.0.0.1:5173`.

### Отдельный запуск CMS API (при необходимости)

```bash
npm run cms:server
```

По умолчанию сервер API слушает `127.0.0.1:8787`.

`npm run preview` теперь тоже прокидывает локальный CMS API middleware (как и `npm run dev`), поэтому форму «Написать нам» можно тестировать прямо на preview-сборке.

## 9. Проверки перед релизом

```bash
npm run build
```

Обязательно проверить вручную:

- главная: `/`
- сведения: `/about`
- документы: `/documents`
- админка: `/admin`
- ключевое видео: `/assets/gimn-ed-zy9mar.mp4`

## 10. Безопасность и секреты

Логины/пароли/кодовые слова CMS не должны храниться в репозитории.

Используем env-файл на сервере, например `/etc/artcomm-cms.env`.

Минимум обязательных секретов:

- `ARTCOMM_CMS_ADMIN_LOGIN`
- `ARTCOMM_CMS_ADMIN_PASSWORD`
- `ARTCOMM_CMS_SECURITY_CODEWORD`
- `ARTCOMM_CMS_PASSWORD_PEPPER`

Пример заготовки:

```bash
sudo install -m 600 -o root -g root deploy/systemd/artcomm-cms.env.example /etc/artcomm-cms.env
```

Рекомендуемые флаги hardening:

- `ARTCOMM_CMS_REQUIRE_ENV_SECRETS=true` (в проде обязательный режим)
- пароли пользователей: минимум 10 символов, обязательно буквы и цифры
- для смены пароля используется дополнительное подтверждение (логин+пароль+кодовое слово)

## 11. Форма «Написать нам»

Форма отправляет данные на `POST /api/cms/contact`.

Режимы работы:

- `ARTCOMM_CONTACT_MODE=smtp` — реальная отправка писем через SMTP,
- `ARTCOMM_CONTACT_MODE=log` — запись заявок в лог (`cms-data/contact-submissions.jsonl`).

Также доступны базовые антиспам-ограничения (лимит/кулдаун по IP).

## 12. Рабочий процесс редактора (без разработчика)

1. Зайти в `/admin`.
2. Внести правки в нужном разделе.
3. Убедиться, что контент корректно отображается на странице.
4. При необходимости открыть «Версионный контроль» и проверить, что версия создана.
5. Нажать «Опубликовать на сайт».
6. Проверить публичные маршруты.

## 13. Типовые проблемы

### Изменение есть в админке, но нет на сайте

- Нажмите «Опубликовать на сайт».
- Обновите страницу жестко (`Cmd + Shift + R`).

### Файл не удаляется

Система не удаляет файл, если он еще используется в `published` или в нескольких местах `draft`.
Это защита от случайной поломки ссылок.

### Не пускает в админку

- Проверьте логин/пароль,
- убедитесь, что CMS API запущен,
- проверьте, что cookie не блокируются браузером.

### Форма «Написать нам» не отправляется

- Проверьте `ARTCOMM_CONTACT_MODE`:
  - в `log` режиме заявки пишутся в `cms-data/contact-submissions.jsonl`,
  - в `smtp` режиме нужны валидные SMTP-параметры.
- Если SMTP не настроен, API вернет `service_unavailable`.

## 14. Деплой через GitHub Actions

В репозитории настроен автоматический деплой из ветки `main` через GitHub Actions:

- workflow: `.github/workflows/deploy.yml`
- при каждом push в `main` сначала запускается сборка
- затем workflow подключается к серверу по SSH и запускает `scripts/deploy-prod.sh`
- вручную тот же workflow можно запустить через `workflow_dispatch`

Что нужно добавить в Secrets репозитория:

- `DEPLOY_HOST` — хост сервера
- `DEPLOY_USER` — пользователь для SSH
- `DEPLOY_SSH_KEY` — приватный SSH-ключ
- `DEPLOY_PORT` — необязательно, если порт не `22`
- `DEPLOY_PATH` — необязательно, если путь не `/var/www/Artcomm`

На сервере должен уже быть развернут проект в `DEPLOY_PATH`, а скрипт `scripts/deploy-prod.sh` — доступен для запуска.
